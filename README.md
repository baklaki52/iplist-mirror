# iplist-mirror

Ежедневное зеркало агрегированного каталога CIDR-диапазонов сервисов
[rekryt/iplist](https://github.com/rekryt/iplist), забираемого через
[iplist.opencck.org](https://iplist.opencck.org) и переупаковываемого
в один JSON-документ с фиксированной схемой и SHA-256-контрольной
суммой — для потребителей, которым нужен воспроизводимый и
верифицируемый источник без обращения к живому API на каждый запрос.

## Что в репозитории

- `snapshot.json` — текущий объединённый каталог (`schema_version=1`)
- `snapshot.sha256` — SHA-256 от `snapshot.json` (hex)
- `snapshot-ru-clean.json` — тот же каталог с вычетом CIDR,
  зарегистрированных за Россией (см. [RU filter](#ru-filter))
- `snapshot-ru-clean.sha256` — SHA-256 от очищенного снимка
- `ru_filter_report.json` — поштучный отчёт: какие префиксы и в каких
  категориях были удалены
- `by-slug.json` — производный плоский формат: карта
  `{slug: [CIDR...]}`, собранная из `snapshot.json`
- `by-slug.json.sha256` — SHA-256 производного файла
- `by-slug-ru-clean.json` — то же, но из очищенного снимка
- `by-slug-ru-clean.json.sha256` — SHA-256
- `all-cidrs.json` — плоский массив всех уникальных CIDR из
  `snapshot.json` (корневой элемент — JSON array, без ключей)
- `all-cidrs.json.sha256` — SHA-256
- `all-cidrs-ru-clean.json` — то же, но из очищенного снимка
- `all-cidrs-ru-clean.json.sha256` — SHA-256
- Git-теги `vYYYYMMDD` — неизменяемые ежедневные срезы
- `.github/workflows/daily-fetch.yml` — расписание сборки
- `scripts/fetch.sh` — однокомандный fetch + нормализация + контрольная сумма
- `scripts/asn_overlay.py` — добавляет в `snapshot.json` BGP-точные CIDR
  для сервисов, чьи DC-подсети не видны через DNS (Telegram); см. [ASN overlay](#asn-overlay)
- `scripts/filter_ru.py` — вычитатель RU-CIDR (двухисточниковый), вызывается из `fetch.sh`
- `scripts/build_flat.py` — сборщик производных форматов `by-slug*.json` и `all-cidrs*.json`

## Схема

```json
{
  "schema_version": 1,
  "generated_at": "2026-04-15T03:00:00Z",
  "source": "rekryt/iplist via iplist.opencck.org",
  "source_tag": "v20260415",
  "services": [
    {
      "slug": "youtube",
      "name": "YouTube",
      "category": "youtube",
      "cidr4": ["142.250.0.0/15"],
      "cidr6": ["2a00::/16"],
      "domains": ["youtube.com"],
      "dns": ["8.8.8.8"]
    }
  ]
}
```

Сервисы отсортированы по `slug`. Категории совпадают с 17 группами
upstream-каталога.

## Как собирается

GitHub Action `daily-fetch` запускается ежедневно в **03:00 UTC**
(а также вручную через `workflow_dispatch`):

1. Забирает 17 категорий с `iplist.opencck.org`
   (`?format=json&group={cat}&data=cidr4&data=cidr6&data=domains&data=dns`).
2. Сливает всё в `snapshot.json` со `schema_version=1`.
3. Считает `snapshot.sha256`.
4. Собирает `snapshot-ru-clean.json` вычитанием российских CIDR
   (см. [RU filter](#ru-filter)).
5. Если относительно предыдущего снимка ничего не изменилось —
   коммит не создаётся.
6. Иначе коммитит все изменённые файлы, ставит тег `vYYYYMMDD`,
   пушит оба.

Concurrency-группа `daily-fetch` предотвращает параллельные запуски.

## RU filter

В upstream-каталоге иногда встречаются CIDR, зарегистрированные за
российскими операторами, внутри категорий зарубежных сервисов
(например, RU-префиксы внутри `youtube` после того как Google вывел
свои GGC-узлы из страны; фрагменты RU CDN в категориях `anime`/`video`).
Такие маршруты для потребителей, маршрутизирующих по категориям,
обычно неработающие — сервис на этих IP либо уже не отвечает, либо
трактует их иначе, чем основную массу каталога.

`scripts/filter_ru.py` выполняется сразу после нормализации в том же
CI-джобе и строит `snapshot-ru-clean.json`:

- **Источники RU-аллокаций (два, объединяются):**
  - [`ipverse/rir-ip`](https://github.com/ipverse/rir-ip) — ежедневные
    срезы, агрегированные из данных RIPE NCC
    (`country/ru/ipv4-aggregated.txt`, `country/ru/ipv6-aggregated.txt`).
    Покрывает всё, что выделено RIPE на Россию по country code.
  - [`russia.iplist.opencck.org`](https://russia.iplist.opencck.org) —
    77 курируемых RU-сервисов (Сбер, Госуслуги, Озон, ВК, Яндекс
    и т.д.) с почасовым DNS re-resolve. Покрывает RU-сервисы, чьи
    IP живут на иностранных ASN (Cloudflare-фронт `.ru` доменов,
    Akamai RU-edge), которые `ipverse/rir-ip` пропускает.
  - **Объединение** этих источников даёт полное покрытие: country-
    уровень + сервис-уровень.
- **Операция:** строгая теоретико-множественная вычитка CIDR.
  Префикс, целиком лежащий внутри RU-блока, удаляется. Префикс,
  целиком содержащий RU-блок, разбивается через
  `ipaddress.IPv4Network.address_exclude` — не-RU территория
  сохраняется. У CIDR-сетей по построению либо пересечение пустое,
  либо одна является подмножеством другой; частичного перекрытия
  не бывает.
- **Дельта схемы:** `snapshot-ru-clean.json` добавляет блок
  `ru_filter` с массивом `sources` (URL и счётчики обоих источников),
  объединёнными счётчиками RU-префиксов и меткой `applied_at`.
  Остальные поля побайтно идентичны `snapshot.json`.
- **Аудит:** `ru_filter_report.json` перечисляет каждый удалённый
  префикс по сервису и категории. Можно сравнивать между тегами
  `vYYYYMMDD` и отслеживать, как меняется upstream-каталог.
- **Режим отказа:**
  - Источник #1 (`ipverse/rir-ip`) недоступен → `filter_ru.py`
    завершается с кодом 0, предыдущий `snapshot-ru-clean.json`
    остаётся на месте.
  - Источник #2 (`russia.iplist.opencck.org`) недоступен → лог
    предупреждения, продолжаем с одним источником #1.
  - Нефильтрованный `snapshot.json` публикуется всегда.

Потребители, которым нужен сырой upstream-каталог, продолжают читать
`snapshot.json`. Потребители, которым нужен очищенный вариант —
читают `snapshot-ru-clean.json` (с проверкой по
`snapshot-ru-clean.sha256`).

## ASN overlay

Upstream-каталог собирается DNS-резолвом тысяч публичных доменов. Но
ряд критичных сервисов имеет выделенные DC-подсети, которых **нет ни в
одной DNS-записи** — DNS-боты резолвинга rekryt их не обнаруживают.

Самый болезненный пример — Telegram. Подсети вида `91.108.36.0/23`,
`91.108.38.0/23`, `95.161.84.0/22` принадлежат AS62041/AS211157 (Telegram
Messenger LLP), реально обслуживают медиа-CDN, но не резолвятся через
`*.telegram.org`. Без них клиентский split-tunnel пропускает трафик
к этим IP мимо VPN — медиа из каналов не загружается через российский
интернет, где конкретные подсети шейпятся или режутся.

`scripts/asn_overlay.py` решает это **до** RU-фильтра:

- **Источник:** канонические BGP-анонсы AS-номеров через
  [`bgpq4`](https://github.com/bgp/bgpq4). Конфигурируется в коде:
  список `OVERLAYS = [{slug, category, asns, min_expected_v4, ...}]`.
- **Операция:** инкрементальная — добавляются только префиксы, которые
  ещё не покрыты существующим набором сервиса. Никогда не удаляет
  upstream-данные.
- **Дельта схемы:** `snapshot.json` получает блок `asn_overlay`
  с `applied_at`, инструментом (`bgpq4`) и поштучным отчётом
  `entries[]` (slug, ASN, сколько v4/v6 добавлено, итоговое количество).
- **Текущая конфигурация:**
  - `telegram.org` ← `AS62041 AS59930 AS44907 AS211157 AS62014`
- **Режим отказа:** если `bgpq4` не установлен, или возвращает
  подозрительно мало префиксов (меньше `min_expected_v4`), или падает
  с ошибкой — overlay пропускается, snapshot остаётся без
  ASN-добавлений. Никаких частичных или сломанных слияний.

Overlay применяется **до** `filter_ru.py`, поэтому ASN-CIDR проходят
через RU-вычитку наравне с upstream-CIDR (если AS62041 случайно
содержит RU-префикс — он будет удалён из чистого снимка).

## Плоские форматы `by-slug` и `all-cidrs`

Рядом с основными снимками публикуются два производных формата для
потребителей, которым не нужно ходить по `services[]`.

### `by-slug.json` — карта по первичному домену

    { "telegram.org": ["1.2.3.0/24", "2001::/32"],
      "whatsapp.com": ["...", "..."] }

Ключ — `service.slug` (в 195 из 201 случаев это пара `domain.tld`),
значение — объединение `cidr4 + cidr6` этого сервиса, без дубликатов,
отсортированное. Подходит когда нужно знать, к какому сервису
относится префикс.

### `all-cidrs.json` — плоский список всех CIDR

    ["1.0.0.0/9", "1.192.0.0/10", "2001:4860::/32", ...]

Корневой элемент — JSON-массив. Это объединение всех `cidr4 + cidr6`
по всем сервисам, без дубликатов, отсортированное. Подходит когда
нужно просто получить набор префиксов (таблицы маршрутизации,
allowlist'ы) — некоторые импортёры требуют именно массив как корневой
элемент, а не объект.

Оба формата собираются для обоих вариантов снимка:

- `by-slug.json` / `all-cidrs.json` — из полного `snapshot.json`
- `by-slug-ru-clean.json` / `all-cidrs-ru-clean.json` — из `snapshot-ru-clean.json`

Каждый файл сопровождается `*.sha256`. Сборкой занимается
`scripts/build_flat.py` — вызывается из `fetch.sh` после `filter_ru.py`.
Ошибка при сборке не блокирует публикацию основных снимков.

Поддомены (`a.b.telegram.org`, `core.telegram.org` и т.д.) в `by-slug.*`
отдельными ключами **не** перечисляются: они разрешаются обычным DNS
в те же адреса из CIDR-пула своего сервиса, поэтому одного ключа на
сервис достаточно.

## Приватность

Каждый запрос на обновление раскрывает IP потребителя для GitHub —
как при обращении браузера за любым публичным файлом. Тем, кому
важны более строгие требования, имеет смысл встраивать снимок
в поставку и обновляться по расписанию, а не по запросу.
